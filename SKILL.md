# JD-Resume Optimizer Skill

## 触发条件

- 用户说"优化简历"、"根据 JD 改简历"、"简历匹配岗位"
- 用户发送岗位 JD 并要求修改简历
- 用户说"帮我看看这个岗位和我的简历匹配度"
- 用户要求"生成 HTML/PDF/截图 简历"

## 支持输入

- **JD**：纯文本 / 截图 / 网页链接
- **简历**：PDF / Word / 纯文本 / 截图

## 输出格式

- HTML 文件（可编辑、可打印）
- PDF 文件（直接投递）
- 长截图（手机查看/转发）

## 核心流程（五步交互）

必须按顺序执行，每步等待用户确认后再进入下一步。

### 第一步：岗位解析

1. **接收 JD**：用户发送纯文本、截图或链接
   - 截图：用户直接粘贴图片即可
   - 链接：使用 `WebFetch` 工具提取网页内容
   - 如果提取失败（如需要登录的页面），提示用户粘贴文本

2. **检查简历状态**：
   - 如果用户之前提供过简历，询问是否继续使用
   - **若 JD 是新的，必须重新索要简历**，避免搞混不同人的简历

3. **关键词提取**（输出给用户看）：
   - 🔧 硬技能：AI 工具、技术栈、方法论（如 Claude Code, LangGraph, AI First）
   - 🧠 软技能：业务 Sense、快速迭代、跨部门协作
   - ⏱️ 经验要求：年限、行业、0-1 经验
   - ⭐ 加分项：独立项目、GitHub、工程背景

4. **岗位本质分析**：用一句话概括这个岗位到底要什么人

### 第二步：简历输入

1. **接收简历**：PDF / Word / 文本 / 截图
   - 首选：让用户直接粘贴简历文本（最可靠）
   - PDF：如果用户上传文件，尝试用 `Read` 或 `Bash`（`pdftotext`）提取文本
   - 截图：用户直接粘贴图片即可

2. **结构化提取**：
   - 头部信息（姓名、联系方式、求职方向）
   - 核心能力/技能
   - 项目经历（名称、技术栈、成果）
   - 工作经历（公司、时间、职责）
   - 教育经历

### 第三步：匹配与调整

1. **匹配度分析**（内部完成，不展示给用户）：
   - 对比 JD 关键词与简历内容
   - 标记：已覆盖 / 角度偏差 / 缺失

2. **微调策略**（关键原则）：
   - ✅ **保留事实**：原有经历的时间、公司、职责不变
   - ✅ **调整叙述角度**：如"数据分析师"→"数据产品化"
   - ✅ **自然融入关键词**：在技能描述中嵌入 JD 关键词
   - ❌ **不编造**：缺失的经历标注"建议补充"，不硬写

### 简历改写十大技巧（必须遵循）

改写简历不是"换几个词"，而是**换一套叙事语法**。以下技巧按优先级排序：

#### 技巧一：叙事角度 — 从"我做了什么"到"我识别了什么痛点"
- **核心原则**：不写岗位职责说明书，写"你在复杂系统里主动发现问题并用一系列动作解决它"
- **对比示例**：
  - ❌ 差写法："负责新客策略开发、优化，参与风控系统搭建。"
  - ✅ 好写法："识别新客风控策略迭代慢的痛点，核心负责准入授信审批全流程策略的设计、上线、监控与迭代，推动内部撞库配置系统从 0 到 1 落地。"
- **效果**：面试官看了会想"这人不是等活干的，是能自己找活干的"

#### 技巧二：动词开头，结果后置
- 每条经历都用**强动词**打头，后面自然带出成果
- **动词强度对照**：
  - ❌ 弱动词：参与、协助、负责相关工作
  - ✅ 强动词：**识别、推动、主导、搭建、设计、验证**
- **示例**：
  - ❌ "使用SQL进行数据处理"
  - ✅ "通过SQL开发中间表优化数据处理流程，缩短报表制作时间30%"

#### 技巧三：关键词自然嵌入，不要硬塞
- JD要求"AI Coding"，不要写"我具备AI Coding能力"——太空
- ✅ **嵌进去**："借助 Claude、Codex 将想法快速转化为可运行的产品，独立完成产品定义→架构设计→前后端联调→上线验证的全链路。"
- **原则**：关键词出现在"我做了什么事"的上下文里，而不是单独列出来

#### 技巧四：项目经历突出"决策链"，不是功能清单
- ❌ **差写法（功能清单）**："项目包含工作台、创建面试、面试房间等模块。"
- ✅ **好写法（决策链）**："从在职跳槽者'不知道薄弱点、缺少追问训练'的痛点出发，设计多 Agent 架构将面试官、追问、评估拆分为独立角色..."
- **区别**：功能清单 = 我做了个APP；决策链 = 我为什么做这个、怎么想的、怎么验证的

#### 技巧五：能力描述用"名词：具体解释"格式
- ❌ 差写法："具备良好的沟通能力和团队协作精神。"
- ✅ 好写法："产品感与业务理解力：从业务视角定义真实问题，将复杂技术方案转化为清晰的业务语言，具备从 0 到 1 推动产品落地的跨部门协作经验。"
- **好处**：先给面试官一个**标签**（好记），再给**证据**（信服）

#### 技巧六：量化但不硬编
- 有数据就放，没数据就不写假数字
- ✅ 你有的："缩短报表制作时间 30%"、"推动 40+ 项风控业务需求落地"、"输出分析报告 10 余篇"
- ❌ 不要编："提升用户留存 200%"（没有真实数据支撑就不写）

#### 技巧七：求职方向直接对标岗位"气质词"
- JD里出现的高频气质词，直接拿来用：
  - JD说"Builder" → 求职方向写"AI Builder"
  - JD说"超级个体" → 能力里写"独立完成全链路"
  - JD说"连接一切" → 经历里写"成为团队中连接一切的人"
- **目的**：让HR在5秒扫描里一眼识别"这人就是我们要的"

#### 技巧八：给面试官"下钩" — 留追问空间
- 写一些有深度但不展开的点，引诱面试官追问：
  - "多 Agent 架构将角色拆分为独立模块" → 面试官会问"怎么拆的？用什么路由？"
  - "模型路由区分复杂推理与轻量任务" → 面试官会问"阈值怎么定的？"
- **这是好事**：简历不是论文，留点钩子，面试时有机会展开讲，显得真有深度

#### 技巧九：去掉"AI味"和"学生味"（一）— 禁用词汇
- **绝对不要出现的词**：
  - "熟练掌握"（换成"熟练使用"或直接写用了什么）
  - "负责相关工作"（换成具体动作）
  - "具备较强的...能力"（换成"做了什么来证明"）
  - "在...方面取得了一定的成绩"（直接说成绩）

#### 技巧十：控制信息密度，一页纸为硬约束
- 超过一页说明优先级判断有问题
- 按岗位相关性排序：最相关的项目/经历放前面，不相关的删掉
- 同一段经历不要罗列超过 5 条职责，面试官只会看前 3 条
- 工作年限短（<3 年）突出项目经历，年限长则突出工作经历

3. **生成优化方案**：准备要修改的具体条目列表

### 第四步：用户确认

向用户展示调整计划（**不包含最终简历文本**，只展示修改计划）：
```
📋 优化方案预览
━━━━━━━━━━━━━━━━━━━━
🔑 JD 关键词：[列表]

📊 匹配度：
   ✅ 已覆盖：X 项
   ⚠️ 角度偏差：X 项（具体说明）
   ❌ 缺失：X 项（建议补充）

📝 计划修改：
   1. [具体修改点]
   2. [具体修改点]
   ...

❓ 确认继续生成？（是/否/调整）
```

- 用户确认 → 进入第五步
- 用户拒绝/调整 → 回到第三步修改方案

### 第五步：最终输出

1. **生成优化后的简历文本**：先以结构化文本展示给用户确认

2. **生成三格式文件**：
   - HTML：使用 `scripts/generate_html_resume.py`
   - PDF：使用 `scripts/html_to_pdf.py`
   - 长截图：使用 `scripts/screenshot_resume.py`

3. **文件输出**：默认保存到 `./output/`
   - 命名：`简历_{姓名}_{岗位关键词}.{格式}`
   - 若路径不可写，提示用户指定输出目录

4. **输出页面要求（必须遵守）**：
   - **去掉生成时间、文件信息**：页面顶部和底部不显示任何元信息
   - **去掉底部链接**：不显示"下载"、"链接"等提示
   - **截图去空白**：使用 PIL 自动裁剪底部空白，精确到内容末尾
   - **排版固定**：使用脚本内置模板（样式与 `assets/resume-template.html` 一致），后续仅允许调整颜色变量，不改布局
   - **生成步骤**：先生成 HTML → 转 PDF → 转截图，不直接给 PDF 截图
   - **PDF 无页眉页脚**：HTML 模板 CSS 中加入 `@page { margin: 0; }`；Chrome 生成 PDF 时使用 `--no-pdf-header-footer` 参数，确保不出现网址、日期、标题等元信息

## 错误处理与兜底

| 场景 | 处理方式 |
|------|---------|
| JD 无法解析（链接失效/图片模糊） | 提示用户直接粘贴文本，跳过关键词提取，由用户手动标注重点 |
| 简历格式混乱，无法结构化 | 让用户分段粘贴（项目经历、工作经历分开），或只优化用户明确指定的部分 |
| JD 与简历完全不匹配 | 如实告知匹配度低，给出"建议补充"清单，不强行改写 |
| 脚本执行失败（Chrome 未安装/PIL 缺失） | 输出 HTML 文件让用户手动打印为 PDF；截图功能降级为不裁剪的原始截图 |

## 重要约束

### 不欺骗原则
- 不编造不存在的经历
- 不夸大未做过的事情
- 技能关键词只在确实掌握的前提下添加
- 对不确定的内容标注为"建议补充"

### 用户偏好（必须遵守）
- ❌ **避免表格形式**：文字直接输出
- ✅ **接地气风格**：增强可信度，反对过于隆重
- ✅ **工作描述范式**："独立负责...覆盖...深入业务前线..."
- ✅ **结构化输出**：用分隔线和 emoji 辅助阅读，但不依赖表格

### 输出风格
- 简历内容用中文
- 分析过程可以活泼，但简历本身保持专业
- 能力描述用「名词 + 冒号 + 具体解释」格式

## 脚本使用

```bash
# 1. 生成 HTML（需要先准备 JSON 数据文件）
python3 scripts/generate_html_resume.py --input resume_data.json --output resume.html

# 2. HTML → PDF
python3 scripts/html_to_pdf.py --input resume.html --output resume.pdf

# 3. HTML → 长截图（精确裁剪，无空白）
python3 scripts/screenshot_resume.py --input resume.html --output resume.png
```

JSON 数据格式参考 `assets/resume-template.html` 中的变量结构。
